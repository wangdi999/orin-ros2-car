# Feature Specification: ROS2 核心与导航四天闭环

**Feature Branch**: `001-ros2-navigation-safety`

**Created**: 2026-07-12

**Status**: Ready for Planning

**Input**: 在受控室内和 ROS 2 Foxy 实车上，用四天、32 小时形成“安全底盘 → 里程计/TF → Cartographer 地图 → AMCL/Nav2 → 三点巡航 → 故障与返航”的可重复验收闭环。

## Clarifications

### Session 2026-07-12

- Q: 本轮验收是否只覆盖仿真或静态检查？ → A: 否，目标为完整实车验收；非运动检查可直接执行，任何首次非零运动测试必须再次取得用户安全批准。
- Q: 是否替换现有底盘驱动？ → A: 复用并加固现有 X3 驱动及 `icar_base_node`，保留现有硬件适配。
- Q: 控制台继续直接写入最终速度话题吗？ → A: 否，控制台改发人工上游话题，最终速度边界仅由仲裁器常规写入。
- Q: 低电返航如何验收？ → A: 默认使用可控模拟触发；真实电压只观察，确认电池规格前不启用真实低电自动返航。
- Q: 测试场地是什么？ → A: 使用已清空或轮子架空的受控室内场地，按低速分级验收。

### Session 2026-07-13

- Q: 后续车端测试连接地址和授权边界是什么？ → A: 当前使用 `192.168.43.137`；除会导致小车移动的测试外，其余车端检查可直接执行。地址属于可变部署配置。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 安全底盘与人工接管 (Priority: P1)

作为现场操作者，我希望所有人工和自主速度请求经过一个可观察、可急停、会在失联时自动归零的安全边界，从而在任何更高层功能失败时仍能可靠停车。

**Why this priority**: 地图和导航都可能发出运动请求；没有独立安全底座，不允许进入任何实车运动阶段。

**Independent Test**: 仅启动底盘、状态估计和安全链路，在轮子架空或场地清空后用最低限速验证人工请求、来源切换、超时、急停、复位和串口断连；不依赖地图或 Nav2 即可交付安全人工控制能力。

**Acceptance Scenarios**:

1. **Given** 底盘与传感器健康且急停未锁存，**When** 操作者给出有限人工速度请求，**Then** 系统标记人工为活动来源，并只向最终硬件边界输出限幅后的有限值。
2. **Given** 人工命令正在生效，**When** 300 ms 内没有新命令，**Then** 最终速度在 0.40 秒内归零并保持为零。
3. **Given** 任意运动请求存在，**When** 急停、串口断连或进程终止发生，**Then** 系统在 0.50 秒内形成零速度、锁存安全状态并发布可诊断告警。
4. **Given** 导航正在请求运动，**When** 人工接管，**Then** 系统先输出零速度、取消当前自主任务，再允许人工控制，且导航不会自行恢复。

---

### User Story 2 - 生成并复用室内地图 (Priority: P2)

作为部署人员，我希望使用稳定的里程计、TF 和激光数据建立可保存、可重载的室内地图，以便后续定位和导航复用同一坐标基准。

**Why this priority**: 地图质量和坐标所有权是定位与路径规划的前置条件。

**Independent Test**: 在安全底座通过后，仅运行建图模式完成一次受控场地采集，保存地图并连续重载两次；无需启动定位或自主导航即可验收。

**Acceptance Scenarios**:

1. **Given** 激光、里程计和固定变换均满足频率与唯一所有者要求，**When** 操作者完成一次室内建图路线，**Then** 系统生成可复用的栅格地图和建图状态文件。
2. **Given** 已保存地图，**When** 地图服务连续加载该地图两次，**Then** 两次均成功且坐标、分辨率和占用信息一致。
3. **Given** 五个预先选定的固定地标，**When** 将地图距离与现场实测比较，**Then** 每个地标误差不超过 0.20 m，且地图没有明显断廊或间距大于 0.15 m 的重复墙。

---

### User Story 3 - 单点自主导航 (Priority: P3)

作为操作者，我希望小车在已知地图中完成一个安全、可取消的目标点导航，并能准确报告成功或失败。

**Why this priority**: 单点导航先证明定位、规划、控制和安全链路闭环，再扩展为多点巡航。

**Independent Test**: 加载已验收地图、设置初始位姿并提交一个可达目标；测量终点位置和朝向，不依赖巡航状态机即可验收。

**Acceptance Scenarios**:

1. **Given** 地图已加载且定位收敛，**When** 提交一个可达目标，**Then** 小车到达目标且位置误差不超过 0.20 m、朝向误差不超过 0.20 rad。
2. **Given** 导航正在执行，**When** 操作者急停、人工接管或取消，**Then** 当前目标立即终止，最终速度归零，并且不会自动继续。
3. **Given** 目标不可达或执行超时，**When** 导航结束，**Then** 系统依据动作终态和本地记录判定失败并给出可追溯原因，而不是误报成功。

---

### User Story 4 - 三点巡航 (Priority: P4)

作为演示人员，我希望从一份可审阅路线中启动三点巡航，在每个点停留，并对不可达点执行一次重试后按策略继续。

**Why this priority**: 这是将单点闭环组合为可重复业务流程的核心演示能力。

**Independent Test**: 使用已验收地图中的三个实测坐标启动非循环路线，观察状态、停留、重试和结束；可单独验证路线解析与状态机，再在实车验证两轮完整执行。

**Acceptance Scenarios**:

1. **Given** 有三个合法航点和 Home 点，**When** 启动巡航，**Then** 系统依次导航、到达、等待 3 秒并进入下一目标，完整路线结束后回到空闲。
2. **Given** 某一航点不可达，**When** 首次导航失败，**Then** 系统只重试一次，仍失败则跳过该点并发布告警，再按配置继续。
3. **Given** 巡航正在执行，**When** 收到取消、人工接管或安全故障，**Then** 当前目标终止，巡航状态退出，恢复后也不会自动续跑。
4. **Given** 三个航点均可达，**When** 连续执行两轮巡航，**Then** 两轮均按顺序完成且每个到达事件有证据记录。

---

### User Story 5 - 故障保护与低电返航 (Priority: P5)

作为安全负责人，我希望关键传感器、速度命令、串口和定位链路故障都能锁止运动并告警，同时可通过模拟低电请求安全返航，以验证异常路径而不依赖真实放电。

**Why this priority**: 完整闭环不仅要在正常路径成功，还必须证明故障时可停车、可诊断、可恢复或安全锁止。

**Independent Test**: 逐项注入急停、命令超时、串口、雷达、里程计/TF、进程退出和模拟低电事件；除返航运动部分外可先完成静态与零速度验证。

**Acceptance Scenarios**:

1. **Given** 系统健康，**When** 注入任一运动相关关键故障，**Then** 0.50 秒内输出零速度、记录结构化告警并进入与故障相符的安全状态。
2. **Given** 故障已锁存，**When** 故障尚未消失或未显式复位，**Then** 任何人工或导航非零请求都不能到达最终速度边界。
3. **Given** 定位有效且 Home 可达，**When** 触发模拟低电，**Then** 当前任务被取消并以受限速度返回 Home；到达后保持停止并告警。
4. **Given** Home 不可达或返航失败，**When** 返航结束，**Then** 系统原地锁止、输出零速度并发布高严重度告警。

### Edge Cases

- 同时收到人工与导航请求时，人工优先；切换来源必须先经历零速度输出周期。
- 急停与来源切换同时发生时，急停优先且锁存，不得因后续健康消息自动清除。
- 速度请求含 NaN、正负无穷或超出硬上限时，异常请求不得传给底盘，并产生诊断记录。
- 串口在驱动启动时缺失、运行中拔出或重新插入时，系统必须保持停车并以固定间隔重连；仅连接恢复不足以解除已锁存安全状态。
- 传感器时间戳倒退、首帧缺少基准或两帧间隔异常时，不得产生里程计跳变。
- 激光 frame 与机器人模型不一致、TF 缺失、TF 重复发布或数据陈旧时，建图与导航门禁必须失败。
- 建图和定位模式被同时请求时，系统拒绝启动第二个地图坐标变换所有者。
- 路线为空、字段缺失、坐标非有限、Home 缺失或失败策略未知时，巡航不得启动。
- 导航动作返回取消、终止、超时或通信丢失时，系统必须使用动作终态判定，不依赖不存在的结果错误码。
- 电压在阈值附近抖动时，真实低电逻辑必须通过滑动平均、持续时间与恢复迟滞避免频繁切换；默认仍不触发真实返航。
- 告警消费者或控制台离线时，底层安全行为不得依赖其在线。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在 ROS 2 Foxy 实车与受控室内场地上支持安全底盘、建图、定位、单点导航、三点巡航、故障保护和返航的顺序验收。
- **FR-002**: 系统 MUST 复用并加固现有 X3 底盘驱动和底盘里程计节点，保持现有串口协议与车体硬件兼容。
- **FR-003**: 所有常规人工与自主速度请求 MUST 进入互相独立的上游通道，且最终硬件速度通道 MUST 只有一个常规发布者；驱动 MUST 在发布者数量或身份不符时拒绝非零命令并停车。
- **FR-004**: 速度仲裁 MUST 使用“急停或不可运动故障高于人工、人工高于普通导航”的优先级，并对活动来源和安全状态提供可观察输出；安全状态心跳陈旧时 MUST fail closed；低电状态是显式例外，只能在新鲜巡航状态同时表明 `mode=RETURN_HOME` 且正在导航时放行返航速度，其他人工或导航请求均保持为零。
- **FR-005**: 来源发生变化时系统 MUST 先输出零速度；人工接管 MUST 取消当前导航或巡航，且不得自动恢复。
- **FR-006**: 底盘驱动 MUST 拒绝非有限速度值，将线速度各轴硬限制在 0.50 m/s 内、角速度限制在 2.00 rad/s 内，并在 300 ms 未收到有效命令时主动停车；首次和分级实车测试仍受 FR-032 的低速门禁约束。
- **FR-007**: 底盘驱动 MUST 在正常终止信号和串口异常时尝试输出零速度；串口断开后 MUST 每 5 秒尝试恢复，并至少以 10 Hz 及状态变化时立即发布连接状态。
- **FR-008**: 状态估计 MUST 提供不低于 20 Hz 的里程计，并防止首帧、异常时间间隔和非有限反馈造成位姿跳变；横移反馈与协方差 MUST 与 X3 实际测量能力一致。
- **FR-009**: 激光数据 MUST 不低于 5 Hz，且传感器 frame MUST 与机器人固定变换中的激光 frame 完全一致。
- **FR-010**: 系统 MUST 维护唯一、连续的 `map → odom → base_footprint → base_link → sensor` 坐标链，并能检测缺失、陈旧或重复所有者；运行期 topic/模式所有权冲突 MUST 锁存为故障、输出零速度并告警。
- **FR-011**: 状态估计 MUST 独占里程计输出和 `odom → base_footprint`；机器人描述 MUST 独占车体到传感器的固定变换。
- **FR-012**: 建图模式和定位模式 MUST 互斥；建图模式与定位模式各自只能有一个 `map → odom` 所有者。
- **FR-013**: 建图 MUST 使用激光和外部里程计，并可保存 `campus_map.pgm`、`campus_map.yaml` 与 `campus_map.pbstream`。
- **FR-014**: 地图 MUST 可由地图服务连续重载两次，并满足五个固定地标误差、走廊连续性和重复墙验收要求。
- **FR-015**: 导航 MUST 使用已验收地图、差速运动约束和零横向自主速度，并在初始实车验收时限制为 0.10 m/s 与 0.40 rad/s。
- **FR-016**: 单点导航 MUST 支持提交、取消、超时、成功和失败，并以目标动作终态、超时和本地重试记录判断结果。
- **FR-017**: 导航目标容差 MUST 配置为 0.15 m 和 0.15 rad，实车结果 MUST 满足 0.20 m 和 0.20 rad 的验收误差。
- **FR-018**: 路线定义 MUST 包含坐标系、Home、航点列表、默认停留秒数、最大重试次数、失败策略和是否循环；坐标 MUST 来源于最终地图实测。
- **FR-019**: 巡航 MUST 明确报告空闲、导航中、已到达、等待和下一目标状态，同时报告 `PATROL`/`RETURN_HOME` 模式、当前目标和终止原因；该状态必须有新鲜度门槛，并支持开始、取消与立即返航。
- **FR-020**: 默认巡航 MUST 在每点停留 3 秒，单点失败最多重试 1 次，重试失败后按 `skip` 策略告警并继续，且默认不循环。
- **FR-021**: 急停、故障、取消或人工接管 MUST 立即终止当前自主目标并阻止自动续跑。
- **FR-022**: 系统 MUST 提供可锁存的急停、安全复位、模拟低电、巡航开始、巡航取消和返航入口；复位只有在触发故障消失时才能成功。
- **FR-023**: 系统 MUST 对急停、串口、雷达、里程计/TF、命令超时、导航失败、低电与进程退出发布带时间、严重度、代码、来源、状态、消息和活动标记的结构化告警。
- **FR-024**: 控制台 MUST 兼容显示结构化告警和现有告警事件流，但告警显示失败不得影响底层停车。
- **FR-025**: 模拟低电 MUST 可独立触发返航流程；真实低电自动返航 MUST 默认关闭。
- **FR-026**: 真实电压观察 MUST 使用最近 10 个样本的平均值，低于 10.8 V 持续 5 秒才满足返航条件，并在高于 11.1 V 后才视为恢复。
- **FR-027**: 返航失败 MUST 使小车原地锁止、保持零速度并发布高严重度告警；不得切回原任务。
- **FR-028**: 控制台 MUST 仅进行人工速度通道、急停/复位入口、ROS 启动编排和相应测试的局部适配，不进行前端结构重写。
- **FR-029**: 每个日门禁 MUST 保存 topic、TF、动作终态、停车延迟、地图质量、故障恢复和验收摘要；原始 rosbag 与大体积日志 MUST 留在本地忽略目录。
- **FR-030**: 所有测试与部署 MUST 保护 `local-config.json`、SSH 凭据、host key 和设备私密信息，并原样保留任务开始前的无关未提交改动。
- **FR-031**: 非运动静态、构建和车端检查 MAY 在当前地址 `192.168.43.137` 上直接执行；任何会发送非零运动命令的测试 MUST 先取得用户对轮子架空或场地清空的明确批准。
- **FR-032**: 第一次运动测试 MUST 使用不超过 0.05 m/s 和 0.20 rad/s；该门禁通过后，后续测试仍不得超过 0.10 m/s 和 0.40 rad/s，且每项运动测试后 MUST 发送零速度。

### Safety & Hardware Constraints *(mandatory for motion-affecting features)*

- 运动请求来源包括控制台/人工遥控、自主导航、巡航/返航编排；速度仲裁器是最终 `/cmd_vel` 的唯一常规发布者，SSH 只保留直接发布零速度的灾难恢复通道。
- 急停和任何运动相关故障均覆盖其他来源并锁存；显式复位必须同时满足故障已消失与当前输出为零。
- 驱动独立执行有限值校验、硬限幅、300 ms watchdog、终止归零与串口断连停车，不依赖控制台或导航层正确工作。
- TF 和 topic 所有权按照 FR-009 至 FR-012 执行；不允许用额外静态 TF 掩盖激光 frame 命名错误。
- 本地单元测试、配置静态检查、网络/端口、topic/参数读取、零速度、构建和故障状态检查属于非运动验证；任何非零 Twist、自动目标或返航都属于物理运动验证。
- 命令超时后最终速度须在 0.40 秒内归零；急停、串口、雷达、里程计/TF、进程退出等运动相关故障须在 0.50 秒内形成零速度和告警。

### Public Interfaces

- **Speed topics**: `/cmd_vel_manual`, `/cmd_vel_nav`, `/cmd_vel`，类型均为 `geometry_msgs/msg/Twist`。
- **State topics**: `/control/active_source`, `/safety/state`, `/patrol/status` 为可读状态；`/patrol/status` 同时是低电返航的 fail-closed 授权/结果握手；`/safety/estop` 为急停输入；`/chassis/connected` 为连接健康状态。
- **Alarm topics**: `/alarm` 提供结构化告警；`/alarm_events` 保留控制台兼容事件。
- **Safety services**: `/safety/reset`, `/safety/simulate_low_battery`。
- **Patrol services**: `/patrol/start`, `/patrol/cancel`, `/patrol/return_home`。
- **Alarm contract**: 每条结构化告警包含 Header、INFO/WARNING/ERROR/CRITICAL 常量以及 `severity`, `code`, `source`, `state`, `message`, `active` 字段。

### Key Entities

- **Motion Request**: 一个带来源、接收时间和有限速度分量的人工或自主请求；由仲裁规则选出或拒绝。
- **Safety State**: 当前是否允许运动、锁存原因、健康前提和复位资格的权威状态。
- **Alarm**: 故障或状态变化的结构化记录，可激活也可清除，并兼容控制台事件流。
- **Transform Owner**: 对一个动态或固定坐标变换负唯一发布责任的运行模式组件。
- **Map Artifact**: 同一次建图会话生成并保持坐标一致的栅格图、元数据和建图状态文件。
- **Route**: 与最终地图绑定的 Home、顺序航点、停留、重试、失败策略和循环选项。
- **Patrol Run**: 一次路线执行的当前目标、状态、尝试次数、到达/失败证据和终止原因。
- **Acceptance Evidence**: 某一门禁的输入条件、观察数据、通过/失败结果、停车延迟与恢复记录。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 控制台修改前的 36 项测试保持全部通过，新增人工通道、急停、复位与启动编排测试也全部通过。
- **SC-002**: 实车激光频率不低于 5 Hz、里程计频率不低于 20 Hz，TF 链无缺口且每个关键变换只有一个所有者。
- **SC-003**: 300 ms 命令超时后，最终速度在 0.40 秒内归零；串口拔插后小车停车、连接状态可见并可恢复。
- **SC-004**: 生成三种 `campus_map` 地图产物，地图服务连续重载两次成功；五个固定地标各自误差不超过 0.20 m，无明显断廊或间距大于 0.15 m 的重复墙。
- **SC-005**: 单点导航终点位置误差不超过 0.20 m、朝向误差不超过 0.20 rad。
- **SC-006**: 三点巡航连续完成两轮；不可达点只重试一次，之后按策略跳过并产生告警。
- **SC-007**: 急停、串口、雷达、里程计/TF、命令超时和优雅终止故障均在 0.50 秒内形成零速度和结构化告警。
- **SC-008**: 模拟低电可取消当前任务并返回 Home；返航失败时保持原地锁止和高严重度告警。
- **SC-009**: 完整一键演示在同一受控室内环境连续成功两次，且每次都有可复核的验收摘要。
- **SC-010**: 车端相关包在目标 Foxy overlay 构建和测试通过，接口检查未发现依赖较新发行版的字段或插件名。

## Assumptions

- 目标仍为车端已安装的 ROS 2 Foxy 环境，本轮不迁移发行版；依赖只使用目标环境已安装且已核验的接口。
- Cartographer 是唯一主线 SLAM；只有其无法达到地图门禁时，才可将 GMapping 作为明确记录的降级方案，且不能宣称主线验收通过。
- 控制台、底盘驱动、激光、rosbridge 和基础机器人模型可在现有代码上增量加固。
- 当前测试地址为 `192.168.43.137`，地址变化属于部署配置，不写入源代码凭据。
- 最终路线坐标和 Home 必须在地图验收后现场录入，仓库初始路线仅提供不可直接运动的占位模板。
- 电池规格尚未确认，所以真实低电只观察；模拟低电是本轮强制验收路径。
- TEB、自动充电、视觉融合、多楼层、GPS、发行版迁移和完整 Web 前端重构不在本轮范围。
