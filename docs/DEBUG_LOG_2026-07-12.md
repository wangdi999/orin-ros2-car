## 3. 问题2：SSH fallback 急停失败 — JSON 转义错误

### 现象

```
[ERROR] [safety] SSH fallback stop failed: Failed to populate field:
'Vector3' object has no attribute 'x:0'
```

### 原因

`server/serviceManager.mjs` 中 `remoteEmergencyStopScript()` 使用 `JSON.stringify` + `replaceAll` 生成急停命令，经过多层 shell 转义（JS 模板 → bash → docker exec bash）后 JSON 格式损坏，`ros2 topic pub` 解析失败。

**原始代码**：
```js
const twist = JSON.stringify(ZERO_TWIST).replaceAll('"', '\\"');
// 生成: {\"linear\":{\"x\":0,...},\"angular\":{...}}
// 经多层 shell 后参数被破坏
```

### 解决

改用 YAML 格式，绕开 JSON 多层转义问题。

**修复后代码**（`server/serviceManager.mjs:391-403`）：
```js
function remoteEmergencyStopScript() {
  // 使用 YAML 格式避免 JSON 在 shell 多层转义中损坏
  const twistYaml = '"{linear: {x: 0.0, y: 0.0, z: 0.0}, angular: {x: 0.0, y: 0.0, z: 0.0}}"';
  return `
set +e
...
docker exec "$cid" bash -lc "... timeout 3 ros2 topic pub --once /cmd_vel geometry_msgs/msg/Twist ${twistYaml}"
`;
}
```

## 4. 问题3：小车不运动 — 底盘驱动反复崩溃

### 现象

控制台操作摇杆，小车不移动。ROSbridge 已连接，`/cmd_vel` 已发布。

### 排查过程

```bash
# 车端检查底盘驱动进程
docker exec smartcar_icar_console ps aux | grep Mcnamu
```

输出：
```
root  218  0.0  0.0  0  0 ?  Z  02:48  0:04 [Mcnamu_driver_X] <defunct>
root  418  0.0  0.0  0  0 ?  Z  02:49  0:06 [Mcnamu_driver_X] <defunct>
root  608  5.0  0.0  0  0 ?  Z  02:51 17:54 [Mcnamu_driver_X] <defunct>
root 27842 0.6  0.0  0  0 ?  Z  08:37  0:02 [Mcnamu_driver_X] <defunct>
root 28751 1.0  0.0  0  0 ?  Z  08:38  0:03 [Mcnamu_driver_X] <defunct>
root 29986 2.7  0.0  0  0 ?  Z  08:39  0:09 [Mcnamu_driver_X] <defunct>
```

**6个僵尸进程**，底盘驱动 `Mcnamu_driver_X3` 反复崩溃。

### 根因

驱动无法访问底盘串口设备（`/dev/myserial` 不存在或已被占用），每次启动后立即崩溃。每次控制台点"启动全部"都在创建新的驱动进程，旧的变成僵尸堆积。

### 诊断命令

```bash
# 查看驱动崩溃日志
docker exec smartcar_icar_console cat /tmp/smartcar_chassis.log

# 检查串口设备
ls -la /dev/myserial /dev/ttyUSB* /dev/ttyACM* 2>/dev/null

# 确认 USB 设备
lsusb
```

### 解决

确认底盘 Type-C 线缆连接 → 确认串口设备出现 → 清理僵尸进程后重启驱动：

```bash
# 杀掉所有驱动僵尸（需重启容器或杀掉父进程）
docker exec smartcar_icar_console pkill -9 -f Mcnamu
# 重新启动
docker exec -d smartcar_icar_console bash -c ". /opt/ros/foxy/setup.bash && . /root/icar_ros2_ws/icar_ws/install/setup.bash && ros2 run icar_bringup Mcnamu_driver_X3"
```

> 用户自行修复（硬件连接问题）。

## 5. 问题4：AI 推理等待图像数据 — Astra 相机 ROS2 驱动缺失

### 现象

小车终端显示"推理线程已启动，等待图像数据"，持续等待无法推理。

### 背景

有两条独立的相机链路：

| 链路 | 用途 | 依赖 |
|------|------|------|
| Python MJPEG (`/dev/video0` → :6500) | 控制台画面 | OpenCV `cv2.VideoCapture(0)` |
| ROS2 `astra_camera` 驱动 | AI 推理图像 topic | ROS2 包及 Orbbec SDK |

控制台画面正常，但 AI 推理没有 ROS2 图像 topic。

### 排查

```bash
# 1. 检查 ROS2 图像 topic
docker exec smartcar_icar_console bash -c ". /opt/ros/foxy/setup.bash && ros2 topic list | grep -iE 'image|color|depth|camera'"
# → 只有 /RGBLight，无图像 topic

# 2. 检查相机节点进程
docker exec smartcar_icar_console ps aux | grep -iE 'astra|camera'
# → astra_camera_no 节点也是僵尸状态

# 3. USB 硬件确认
lsusb | grep 2bc5
# → 050f 和 060f 均已识别

# 4. 检查 ROS2 包
docker exec smartcar_icar_console bash -c ". /opt/ros/foxy/setup.bash && ros2 launch astra_camera astra.launch.xml"
# → Package 'astra_camera' not found
```

### 根因

Docker 容器内未安装 Orbbec Astra 相机的 ROS2 驱动包。

### 解决方向

方法1（优先尝试）：
```bash
docker exec -it smartcar_icar_console bash -c "apt update && apt install -y ros-foxy-astra-camera"
```

方法2（源码编译）：
```bash
docker exec -it smartcar_icar_console bash
mkdir -p /root/ros2_astra_ws/src && cd /root/ros2_astra_ws/src
git clone https://github.com/orbbec/OrbbecSDK_ROS2.git -b foxy
cd /root/ros2_astra_ws
. /opt/ros/foxy/setup.bash
rosdep install --from-paths src --ignore-src -y
colcon build
. install/setup.bash
ros2 launch orbbec_camera astra.launch.xml
```

### 8.2 相机管线问题深入诊断

**现象**：`/camera/color/image_raw` 话题不存在，AI 推理节点阻塞在"等待图像数据"。

**根因分析**：

1. **MJPEG 服务器**：主机 PID 4095 (python3) 独占 `/dev/video0` 并在 6500 端口提供 MJPEG 服务
2. **MJPEG 流格式**：`http://localhost:6500/` 返回 HTML 页面，实际视频流在 `/video_feed`
3. **流确认**：`/video_feed` 返回 HTTP 200，`Content-Type: multipart/x-mixed-replace; boundary=frame` — 是标准 MJPEG 流
4. **OpenCV 兼容性**：`cv2.VideoCapture("http://localhost:6500/video_feed")` 无法正确打开（V4L2 后端冲突），但 `requests` + `cv2.imdecode` 可以解析
5. **Docker 内相机**：Docker 容器也无法通过 OpenCV 打开 `/dev/video0`（设备被主机 MJPEG 服务锁定）

**解决方案**：创建 `mjpeg_bridge.py` 节点，使用 `requests` 库读取 MJPEG 流 → `cv2.imdecode` 解码 → 发布到 `/camera/color/image_raw`

### 8.3 mjpeg_bridge 节点方案

- 使用 `requests.get(url, stream=True)` 读取 MJPEG multipart 流
- 按 boundary 分割提取 JPEG 帧
- `cv2.imdecode` 解码为 numpy 数组
- `cv_bridge` 转为 ROS2 Image 消息发布
- 入口点：`ros2 run car_ai_vision mjpeg_bridge`（已编译在 `install/bin/`）

## 10. 第二轮修复：MJPEG流损坏 + numpy兼容性 + spin_once（2026-07-12 20:30）

### 10.1 MJPEG流诊断

**现象**：桥接成功连接 `/video_feed` 但 `ros2 topic hz` 显示0数据。
**根因**：`curl` 下载 `/video_feed` 返回 0 字节——MJPEG服务器(PID 4095)虽响应HTTP 200但流已损坏。

### 10.2 最终方案：直接OpenCV

1. 杀掉PID 4095释放 `/dev/video0`
2. `start_camera.py`：OpenCV直接打开相机 → 发布ROS2话题
3. 关键修复：**`rclpy.spin_once(node, timeout_sec=0.001)`**——没有这一步，`pub.publish()` 的消息永远停留在DDS队列

### 10.3 numpy/TensorRT兼容性

TensorRT 8.5使用 `np.bool`，numpy>=1.24已移除。修复：`np.bool = bool`（`yolov8_inference.py`第34行）。

### 10.4 FPS阈值调整

Jetson Orin Nano 4GB实测~12FPS（YOLOv8s TensorRT FP16），告警阈值从15降至10。

### 10.5 固定启动流程
