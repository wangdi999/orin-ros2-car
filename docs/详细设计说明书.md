### 2.2 yolov8_inference.py — YOLOv8 推理主节点

#### 2.2.1 节点: YOLOv8InferenceNode

**类继承**: `rclpy.node.Node`

**三线程架构**:

| 线程 | 职责 |
|------|------|

| `self._odom_lock: threading.Lock` | `_latest_odom_msg` 里程计缓冲区 |
| `self._tf_lock: threading.Lock` | `_latest_tf_msgs` TF 变换缓冲区 |

#### 2.2.6 推理主循环算法 (`_inference_loop`)

```
LOOP:
  1. event.wait() — 阻塞等待新帧信号
  2. event.clear()
  3. 加锁拷贝 _latest_color_msg, _latest_depth_msg, _latest_odom_msg
  4. 校验: 空数据 → skip; 编码非 rgb8/bgr8 → error → skip
  5. cv_bridge 转换 → np.ndarray (BGR)
  6. model(image, conf=0.5, classes=[0]) — YOLOv8 推理
  7. 提取检测: bboxes, confidences (仅 COCO person=0)
  8. 无检测 → 发布空可视化帧 → continue
  9. 深度关联: _extract_depth_values(bboxes)
     - 对每个 bbox 计算 ROI 深度中位数
     - 无效/越界 → None
  10. 异常行为判定: _abnormal_detector.update(bboxes, depth_values)
     - 返回 bool 列表，逐框标记异常
  11. 获取当前位置: _get_current_position()
     - 优先: odom → map (TF变换)
     - 降级: odom 坐标 (0,0) + "odom" 坐标系
  12. 逐目标处理:
      FOR EACH (bbox, confidence, is_abnormal):
        a. danger_type = "abnormal_behavior" if is_abnormal else "person_detected"
        b. 消抖检查: _debouncer.should_publish(danger_type)
        c. 发布报警: _publish_alarm(danger_type, confidence, pos, coord_frame, bbox)
           - 构造 Alarm 消息
           - validate_alarm_data() 全字段校验
           - 发布到 /cassis/ai_alarm
           - 异常行为 → capture_manager.trigger()
        d. capture_manager.feed(frame) 持续缓冲
  13. 可视化: visualizer.draw_detections(image, bboxes, confs, abnormals)
      - 正常: 绿框 (0,255,0), 标签 "person 0.XX"
      - 异常: 红框 (0,0,255), 闪烁 5帧周期, 标签 "ABNORMAL 0.XX"
      - 叠加 FPS + 推理延迟 + 检测数
  14. _publish_visualization(annotated_image, header)
  15. _log_performance_stats() — 每 5 秒输出统计
      - FPS, 平均推理延迟(ms), 处理帧数, 报警发布数
      - FPS < 10: WARN; 延迟 > 60ms: WARN
```

| `validate_confidence` | `(conf: float) -> bool` | `isinstance(conf, (int, float)) and 0.0 <= conf <= 1.0 and not isnan(conf) and not isinf(conf)` |

#### 2.2.8 坐标变换 (`_transform_position`)

```
输入: odom_x, odom_y, odom_z=0.0
输出: (map_x, map_y, coord_frame)

1. 遍历 _latest_tf_msgs 缓冲区
2. 匹配 child_frame_id="odom" AND frame_id="map"
3. 四元数旋转 + 平移变换
4. 成功 → (map_x, map_y, "map")
5. 失败 → (odom_x, odom_y, "odom")  # 降级
```

#### 2.2.9 深度值提取 (`_extract_depth_values`)

```
FOR EACH bbox (x1, y1, x2, y2):
  1. 对 bbox 内区域取深度图 ROI
  2. 展平后取 np.median()
  3. 校验: 值在 (0.1, 30.0) 米范围内
  4. 合法 → median, 非法 → None
```

### 2.3 abnormal_behavior.py — 异常行为检测器

#### 2.3.1 类: AbnormalBehaviorDetector

**设计原理**: 基于规则的 AND 条件判定倒地行为，避免单一特征误判。

```python
class AbnormalBehaviorDetector:
    def __init__(self, consecutive_frames: int = 15, iou_threshold: float = 0.5)
```

**四个判定条件 (全部 AND)**:

| 条件 | 公式 | 阈值 | 说明 |
|------|------|------|------|
| C1: 宽高比 | `width / max(height, 1e-6) > 1.5` | > 1.5 | 倒地姿态，正常站立 0.3~0.6 |
| C2: 连续静止 | IoU(prev_bbox, curr_bbox) ≥ 0.5 × N帧 | N=15 | 追踪同一人不移动 |
| C3: 深度方差 | `np.var(depth_history) < 0.3` | < 0.3 | 全身紧凑，排除行走 |
| C4: 滞回锁 | 恢复: `height > standing_height × 0.8` | > 0.8 | 防止误恢复 |

#### 2.3.2 追踪数据结构

```python
_tracks[track_id] = {
    "bbox_history": [(x1,y1,x2,y2), ...],  # 最近 N 帧
    "depth_history": [float, ...],
    "standing_height": float | None,        # 最大站立高度
    "fallen": bool,                          # 当前倒地状态
    "still_count": int,                      # 连续静止帧数
    "stale_count": int,                      # 失活帧数 (>60 清理)
}
```

#### 2.3.3 update() 算法流程

```
输入: bboxes: List[tuple], depth_values: List[float|None]
输出: List[bool] — 与 bboxes 一一对应

FOR EACH track:
    track["stale_count"] += 1

FOR EACH bbox IN bboxes:
    1. track_id = _match_track(bbox)  # IoU ≥ 0.5 匹配已有跟踪 OR 新建
    2. track["stale_count"] = 0
    3. C1: aspect_ratio = width / height
    4. 更新 bbox_history (维护滑动窗口)
    5. C2: 计算当前与上一帧 IoU
       - IoU ≥ 0.5 → still_count += 1
       - 否则 → still_count = max(0, still_count - 1)
    6. is_still = still_count ≥ consecutive_frames
    7. C3: 若 depth_values 存在则计算方差
       - 无深度数据 → depth_ok = True (跳过)
    8. 记录站立高度 (非倒地 + 非宽高比异常时)
    9. C4 滞回:
       IF fallen:
           IF height/standing_height > 0.8 → fallen = False, reset
           ELSE → is_abnormal = False
       ELSE:
           is_abnormal = is_wide_aspect AND is_still AND depth_ok
           IF is_abnormal → fallen = True
    10. results.append(is_abnormal)

清理: stale_count > 60 的追踪删除
```

### 2.4 alarm_manager.py — 报警消抖器

#### 2.4.1 类: AlarmDebouncer

**功能**: 每个 danger_type 独立维护状态机，防止同类型报警在短时间内重复发布。

**状态枚举**:

```python
class AlarmState(Enum):
    IDLE      = "IDLE"        # 空闲，可触发新报警
    DETECTED  = "DETECTED"    # 本周期已触发
    COOLDOWN  = "COOLDOWN"    # 冷却期，禁止发布
    RECOVERY  = "RECOVERY"    # 恢复观察期 (2s)
```

**状态转换图**:

```
IDLE ──[检测到]──> DETECTED ──[下一周期]──> COOLDOWN
  ↑                                          │
  │                                    [冷却时间到]
  │                                          ↓
  └────[恢复期满]──── RECOVERY ◄──────────────┘
                        (观察 2s)
```

**冷却配置 COOLDOWN_CONFIG**:

| danger_type | 冷却时间 (秒) |
|-------------|--------------|
| `person_detected` | 30 |
| `abnormal_behavior` | 10 |
| `cracked_tile` | 30 |

> 异常行为的冷却时间 (10s) 比人员检测 (30s) 更短，确保倒地等紧急事件能更快重新触发。

#### 2.4.2 核心方法

```python
def should_publish(self, danger_type: str) -> bool:
    """驱动状态机，返回是否允许本次发布"""
    state = self._states[danger_type]
    now = time.time()
    last = self._timestamps[danger_type]

    if state == AlarmState.IDLE:
        if now - last >= self.get_cooldown(danger_type):
            self._states[danger_type] = AlarmState.DETECTED
            return True
    elif state == AlarmState.DETECTED:
        self._states[danger_type] = AlarmState.COOLDOWN
        self._timestamps[danger_type] = now
    elif state == AlarmState.COOLDOWN:
        if now - last >= self.get_cooldown(danger_type):
            self._states[danger_type] = AlarmState.RECOVERY
            self._timestamps[danger_type] = now
    elif state == AlarmState.RECOVERY:
        if now - last >= 2.0:  # 2秒观察期
            self._states[danger_type] = AlarmState.IDLE
    return False
```

```
1. EMERGENCY_STOP
   IF emergency_stopped → ("EMERGENCY_STOP", Velocity())

2. MANUAL_TELEOP
   IF teleop_at_ms exists AND (now_ms - teleop_at_ms) <= teleop_timeout_ms
      → ("MANUAL_TELEOP", teleop)

3. NAVIGATION
   IF patrol_running
      AND navigation_at_ms exists
      AND (now_ms - navigation_at_ms) <= navigation_timeout_ms
      → ("NAVIGATION", navigation)

4. ZERO (fallback)
   → ("ZERO", Velocity())
```

**关键设计决策**:
- 急停是最低层级的锁存信号，优先级最高
- 导航仅在巡逻运行中生效，防止未授权自动行驶
- 超时窗口使用 `<=` (边界不算超时)，防止边缘抖动

  1. 获取当前时间戳 monotonic_ms()
  2. 调用 choose_velocity() 进行仲裁
  3. 调用 sanitize() 进行限幅

  5. 若源切换 → 发布 /safety/state JSON

##### 急停处理

```python
def _on_emergency_stop(self, msg: Bool):
    if msg.data:  # 激活急停
        self._emergency_stopped = True
        self._teleop_at_ms = None       # 丢弃所有源时间戳
        self._navigation_at_ms = None
        self._publish_zero_burst()      # 连续 3 次零速度
    else:
        self._emergency_stopped = False  # 明确清除 (不自动恢复)

def _publish_zero_burst(self):
    """连续发送 3 次零 Twist，间隔 80ms"""
    for _ in range(3):
        self._cmd_vel_pub.publish(Twist())
        time.sleep(0.08)

def destroy_node(self):
    self._publish_zero_burst()  # 销毁前确保停车
    super().destroy_node()
```

#### 3.3.1 task_state.py — 任务状态机

##### PatrolState 枚举

```python
class PatrolState(str, Enum):
    IDLE              = "IDLE"               # 空闲，无活动任务
    READY             = "READY"              # 任务已创建，等待启动
    RUNNING           = "RUNNING"            # 正在导航
    PAUSED            = "PAUSED"             # 暂停
    SUCCEEDED         = "SUCCEEDED"          # 所有路径点完成 (终态)

    CANCELLED         = "CANCELLED"          # 外部取消 (终态)
    RECOVERY_REQUIRED = "RECOVERY_REQUIRED"  # 等待人工干预
```

**终态集合**: `{SUCCEEDED, FAILED, CANCELLED}`
**活动状态 (阻止新任务)**: `{READY, RUNNING, PAUSED, RECOVERY_REQUIRED}`

##### PatrolTask 状态转移表

| 操作 | 前置状态 | 目标状态 |
|------|----------|----------|
| `start()` | READY | RUNNING |
| `pause()` | RUNNING | PAUSED |
| `resume()` | PAUSED | RUNNING |
| `cancel()` | READY/RUNNING/PAUSED/RECOVERY_REQUIRED | CANCELLED |
| `waypoint_succeeded()` | RUNNING | RUNNING 或 SUCCEEDED |
| `waypoint_failed()` | RUNNING | RUNNING 或 RECOVERY_REQUIRED |
| `retry_current()` | RECOVERY_REQUIRED | RUNNING |
| `skip_current()` | RECOVERY_REQUIRED | RUNNING 或 SUCCEEDED |

**重试逻辑**: `retry_count` 默认最大=1，超过 → `RECOVERY_REQUIRED`。成功后重置计数器。

##### 导航循环 (`_send_current_goal`)

```
1. 等待 Nav2 Action Server 就绪 (timeout=200ms)
   → 超时 → RECOVERY_REQUIRED
2. 从 LocationStore 获取航点 Pose
3. 发送导航目标: NavigateToPose.Goal()
4. 异步回调:
   - _on_goal_response: 确认目标被接受
   - _on_goal_result:
     - 成功 → waypoint_succeeded() → current_index++
       - 所有航点完成 → SUCCEEDED
       - 否则 → _send_current_goal() 递归
     - 失败 → waypoint_failed()
       - 有重试配额 → schedule_once(retry_delay_sec) → _send_current_goal()
       - 重试耗尽 → RECOVERY_REQUIRED
```

##### 目标令牌机制 (goal_token)

为防止异步回调错乱 (取消旧目标后旧结果仍可能触发回调)，每次发送新目标时递增 `goal_token`，回调中校验 `token == goal_token`。
PAUSE/CANCEL 时递增 token 使旧目标回调失效。

### 6.2 Mcnamu_driver_X3 — 底盘驱动

**节点**: `mcnamu_driver`
**订阅**: `/cmd_vel` (`geometry_msgs/Twist`)
**发布**: `/vel_raw` (`geometry_msgs/Twist`), `/joint_states` (`sensor_msgs/JointState`), `/voltage` (`std_msgs/Float32`)

**Mecanum 运动学解算**:
```
v_front_left  = +linear.x - linear.y + angular.z * (wheelbase_x + wheelbase_y)
v_front_right = +linear.x + linear.y + angular.z * (wheelbase_x + wheelbase_y)
v_rear_left   = +linear.x + linear.y - angular.z * (wheelbase_x + wheelbase_y)
v_rear_right  = +linear.x - linear.y - angular.z * (wheelbase_x + wheelbase_y)
```

## 7. 异常处理策略

### 7.1 AI 推理层

| 异常 | 处理 |
|------|------|
| 模型加载失败 | 按 .engine → .onnx 降级，均失败则异常退出 |
| 推理 OOM (GPU MemoryError) | 清理 GPU 缓存，跳过当前帧继续 |
| 单帧解码失败 | 跳过该帧，继续处理下一帧 |
| 里程计超时 (5s) | 坐标降级为 (0, 0) + "odom" 坐标系 |
| TF 变换不可用 | 坐标降级为 odom 坐标系 |
| 深度数据不可用 | 跳过深度校验条件 (C3)，其他条件继续判定 |

### 7.2 安全仲裁层

| 异常 | 处理 |
|------|------|
| NaN/Inf 速度分量 | `sanitize()` 返回 None → 不发布 → 保持上一次有效值 |
| 手柄超时 (>450ms) | 降级到 Nav2 或零速 |
| 导航超时 (>500ms) | 降级到零速 |
| 急停激活 | 锁存 + 清空所有源时间戳 + 零速脉冲 |
| 节点销毁 | 发送零速脉冲后退出 |

### 7.3 Agent 编排层

| 异常 | 处理 |
|------|------|
| LLM 调用失败 | `llm_status=DEGRADED`，回退到 MockPlanProvider |
| 计划校验有错误 | 路由到 `failed` 节点，task_state=REJECTED |
| 网关返回错误 | 设置 `error_code/error_message`，更新任务状态 |
| 未授权 API 请求 | HTTP 401, 不泄露路径信息 |

### 7.4 控制台层

| 异常 | 处理 |
|------|------|
| rosbridge 断连 | 自动重连 (2500ms 延迟) |
| 驾驶看门狗超时 | 自动急停 |
| WebSocket 断连 | 自动急停 |
| SSH fallback 急停失败 | 记录错误日志 |
| Plink SSH 超时 | `{ok: false, timedOut: true}` |
